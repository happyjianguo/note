元数据

定义：描述数据属性的数据。在数据仓库系统中，元数据可以帮助数据仓库管理员和数据仓库的开发⼈员⾮常⽅便地找到他们所关⼼的数据。可按其⽤途分为技术元数据和业务元数据。

元数据分类

技术元数据

技术元数据是存储关于数据仓库系统技术细节的数据，是⽤于开发和管理数据仓库使⽤的数据，包括：

  1. 数据仓库结构的描述，模式、试图、维、层次结构和导出数据的定义，数据集市的位置和内容；
  2. 业务系统、数据仓库和数据集市的体系结构和模式；
  3. 汇总⽤的算法，包括度量和维定义算法，数据粒度、主题领域、聚集、汇总、预定义的查询与报告；
  4. 由操作环境到数据仓库环境的映射，包括源数据和它们的内容、数据分割、数据提取·、清理、转换规则和数据刷新规则

业务元数据

业务元数据从业务⻆度描述了数据仓库中的数据，它提供了介于使⽤者和实际系统之间的语义层，包括：

  1. 企业概念模型：表示企业数据模型的⾼层信息、整个企业的业务概念和相互关系；
  2. 多维数据模型：企业概念模型的重要组成部分，告诉业务分析⼈员在数据集市当中有哪些维、维的类别、数据⽴⽅体以及数据集市中的聚合规则；
  3. 业务概念模型和物理数据之间的依赖：业务视图与实际的数据仓库或数据库、多维数据库中的表、字段、维、层次等之间的对应关系

元数据价值

元数据是数据管理、数据内容、数据应用的基础。

在数据管理方面为集团数据提供在计算、存储、成本、质量、安全、模型等治理领域上的数据支持；

在数据内容方面为集团数据进行数据域、数据主题、业务属性等的提取和分析提供数据素材。例如利用元数据构建知识图谱，给数据打标签，清楚知道现在有哪些数据；

在数据应用方面打通产品及应用链路，保障产品数据准确、及时铲除。

元数据管理平台

元数据管理包括业务词汇表的发展，数据元素和实体的定义，业务规则和算法以及数据特征。最基础的管理是管理业务元数据的收集、组织和维持。

元数据管理起到承上启下的作⽤

  1. 元数据是进⾏数据集成所必须的

    数据仓库最⼤的特点是集成性。这⼀特别不仅要体现在它所包含的数据上，还体现在实施数据仓库项⽬的过程中。⼀⽅⾯，从各个数据源中抽取的数据要按照⼀定的模式存⼊数据仓库中，这些数据源与数据仓库中数据的对应关系及转换规则都要存储在元数据知识库中；

    另⼀⽅⾯，在数仓项⽬实施过程中，直接建⽴数据仓库往往费时费⼒，因此在实践过程中，⼀般会按照统⼀的数据模型，⾸先建设数据集市，然后在各个数据集市的基础上再建设数据仓库。当数据集市的数量增多时很容易形成“蜘蛛⽹”现象，元数据管理正是解决这种现象的关键。

  2. 元数据定义的语义层可以帮助最终⽤户理解数据仓库中的数据

    元数据可以实现业务模型与数据模型之间的映射，因⽽可以把数据以⽤户需要的⽅式“翻译”出来，从⽽帮助⽤户理解和使⽤数据仓库。

  3. 元数据是保证数据质量的关键

    借助元数据管理系统，使⽤者对各个数据的来⻰去脉以及数据抽取和转换的规则都会很⽅便的看到元数据可以⽀持需求变化成功的元数据管理系统可以把整个业务的⼯作流、数据流和信息流有效的管理起来，使得系统不依赖特定的开发⼈员，从⽽提⾼系统的可扩展性

元数据收集

对于大数据开放平台来说，常见的元数据信息包括：

  • 数据的表结构Scheme信息
  • 数据的存储空间、读写记录、权限归属和其他各类统计信息
  • 数据的血缘关系信息
  • 数据的业务属性信息

数据的表结构Scheme信息

虽然数据存储组件多半自身都有管理和查询表结构Schema的能力，但是本质上这些系统自身的元数据信息管理手段，通常都是为了满足系统自身的功能运转而设计的，并不是为了数据质量管理的目的而存在的，通常无法直接满足数据质量管理的需求。

收集结构信息，不仅是简单的信息汇总，更重要的是从平台管理和业务需求的角度来考虑如何整理和归纳数据，方便系统集成，实现最终的业务价值。

数据的存储空间、读写记录、权限归属和其他各类统计信息

可能包括但不限于：数据占据了多少底层存储空间，最近是否有过修改，都有谁在什么时候使用过这些数据，谁有权限管理和查阅这些数据等。

对于数据质量管理，这些信息对于系统和业务的优化、数据的安全管控、问题的排查等工作来说，往往都是不可或缺的重要信息。

数据的血缘关系信息

当数据出现问题，可以根据血缘关系往上有排查；

也可以通过数据的血缘关系建立起生产这些数据的任务之间的依赖关系，进而辅助调度系统的工作调度，或者用来判断一个失败或错误的任务可能对哪些下游数据造成影响等。

数据的业务属性信息

常见的比如一张数据表的统计口径信息，这张表的用处、各个字段的具体统计方式、业务描述、业务标签、脚本逻辑的历史变迁记录、变迁原因等。此外用户也可能会关心对应的数据表格是由谁负责开发的，以及具体数据的业务部门归属等。

这部分信息通常与底层系统自身的运行逻辑无关，往往需要通过外部手段人工录入，但是还需要尽量考虑和流程进行整合，让它们称为业务开发必不可少的环节。

元数据管理的标准化

随着元数据联盟（ Meta Data Coalition）的开放信息模型OIM（ Open information Model）和OMG组织的公共仓库模型CWM（ Common Warehouse Model）标准的逐渐完善，以及MDC和OMG组织的合并，为数据仓库⼚商提供了统⼀的标准，从⽽为元数据管理铺平了道路。

  1. 元数据管理主要有两种：

    对于相对简单的环境，按照通⽤的元数据管理标准，建⽴⼀个集中式的元数据知识库;

    对于复杂的环境，分别建⽴各部分的元数据管理系统，形成分布式元数据知识库，通过建⽴标准的元数据交换格式，实现元数据的集成管理

  2. MDC的OIM存储模型

    OIM的⽬的是通过公共的元数据信息来⽀持不同⼯具和系统之间数据的共享和重⽤。 OIM所声明的元数据类型都采⽤统⼀建模语⾔UML（ Universal Modeling Language）进⾏描述，并被组织成易于使⽤、易于扩展的多个主题范围，这些主题范围包括：

    分析与设计

    主要⽤于软件分析、设计和建模。

    对象与组件

    涉及⾯向对象开发技术的⽅⽅⾯⾯。

    数据库与数据仓库

    为数据库模式管理、复⽤和建⽴数据仓库提供元数据概念⽀持。

    业务⼯程

    为企业运作提供⼀个蓝图。

    知识管理

    涉及企业的信息结构。

  3. OMG的CWM模型

    公共仓库元模型的主要⽬的是在异构环境下，帮助不同的数据仓库⼯具、平台和元数据知识库进⾏元数据交换。基于以下

    三个⼯业标准制定：

    UML

    对CWM模型进⾏建模

    MOF（元对象设施）

    是OMG元模型和元数据的存储标准，提供在异构环境下对元数据知识库的访问接⼝

    XMI（ XML元数据交换）

    可以使元数据以XML⽂件流的⽅式进⾏交换

  4. CWM模型为数据仓库和商业智能⼯具之间共享元数据，制定了⼀整套关于语法和语义的规范，这套规范包括四个⽅⾯：

    CWM元模型

    描述数据仓库系统的模型

    CWM XML

    CWM元模型的XML表示

    CWM DTD

    DW/BI共享元数据的交换格式

    CWM IDL

    DW/BI共享元数据的应⽤程序访问接⼝

  5. CWM和OIM之间的关系

    CWM实际上是专⻔为数据仓库元数据⽽制定的⼀套标准，⽽OIM并不是针对数据仓库元数据的;

    CWM只限定于数据仓库，⽽OIM模型包括：分析与设计模型、对象与组件、数据库与数据仓库、商业⼯程、知识管理等五个领域;

    在建模语⾔的选择、数据库模型的⽀持、 OLAP分析模型的⽀持、数据转换模型的⽀持⽅⾯都⽐较⼀致;

    OIM并不是基于元对象设施的，这意味着⽤OIM所描述的元数据需要通过其他的接⼝才能访问，⽽CWM所描述的元数据可以通过CORBA IDL来访问;

    在数据交换⽅⾯， OIM必须通过特定的转换形成XML⽂件来交换元数据，⽽CWM可以通过XMI来进⾏交换

元数据管理系统的设计原则

当前可以认为OMG的CWM将会成为数据仓库元数据领域事实上的标准。建⽴元数据管理系统时，绝对不能盲⽬追求⼤⽽全，要坚持⽬标驱动的原则，在实施的时候采⽤增量式、渐进式的建设原则。

  1. 建设数据仓库系统初期，⾸先要确定系统的边界范围，系统范围确定的原则是⾸先保障重点
  2. 系统边界确定以后，把现有系统的元数据整理出来，加⼊语义层的对应，然后存到⼀个数据库中
  3. 确定元数据管理的范围，⽐如想通过元数据来管理数据仓库中数据的转换过程以及有关数据的抽取路线
  4. 确定元数据管理的⼯具，采⽤⼀定的⼯具可以完成相应的⼯作，如：微软的Repositry，它带有相应的编程接⼝，可以借助于它来完成元模型出⼊库的功能；与之相似的还有Platinum的OEE；另外还有Sybase的Wcc，它可以通过MDC以前的⼀个⽼标准――MDIS来集成抽取⼯具与转换⼯具，在⼀个窗⼝中就可以表示数据抽取与转换，并且可以把语义层以MDIS的格式导出到⼀个前端⼯具当中（⽐如Cognos的Improptu）

主数据和主数据管理的概念

  1. 企业主数据是⽤来描述企业核⼼业务实体的数据。是具有⾼业务价值、可以在企业内跨越各个业务部⻔被重复使⽤的数据，存在于多个异构的应⽤系统中。
  2. 数据管理的范畴和主数据管理的概念

    企业数据管理的内容及范畴通常包括交易数据、主数据以及元数据。

  3. 主数据管理的意义

    主数据管理的四⼤要素：集成、共享、数据质量、数据治理。

    主数据要做的就是从企业的多个业务系统中整合最核⼼的、最需要共享的数据（主数据），集中进⾏数据的清洗和丰富，并以服务的⽅式把统⼀的、完整的、准确的、具有权威性的主数据分发给全企业范围内需要使⽤这些数据的操作型应⽤和分析型应⽤，包括各个业务系统、业务流程和决策⽀持系统等。

    主数据管理使得企业能够集中化管理数据，在分散的系统间保证主数据的⼀致性。从IT建设的⻆度，主数据管理可以增强IT结构的灵活性，构建覆盖整个企业范围内的数据管理基础和相应规范，并且更灵活地适应企业业务需求的变化。

    实现SOA的体系结构：建⽴主数据系统之前，数据被锁定在每⼀个应⽤系统和流程中，建⽴主数据管理系统之后，数据从应⽤系统中释放出来，并且被处理成为⼀组可重⽤的服务，被各个系统调⽤。

主数据管理系统与数据仓库系统的关系

  1. 相同点

    都是跨部⻔的集中式系统，对企业具有相同的价值，可以减少数据冗余和不⼀致性、提升对数据的洞察⼒

    都依赖很多相同的技术⼿段，都涉及ETL，都需要元数据管理、都强调数据质量

    建设⼿段类似，都需要数据治理的规范作为指导，都需要不同系统、不同部⻔的协作，需要统⼀的安全策略

  2. 不同点

    处理类型不同:主数据管理系统是便交易型的系统，为各个业务系统提供联机交易服务，系统的服务是呼叫中⼼、 B2C、 CRM等业务系统,数据仓库是属于分析型的系统，⾯向的是分析型的应⽤，是在⼤量历史交易数据的基础上进⾏多维分析，系统的使⽤对象是各层领导、业务分析、市场销售预测⼈员

    实时性不同:与传统的数据仓库⽅案的批量ETL⽅式不同，主数据管理系统在数据初始加载阶段要使⽤ETL，但在后续运⾏中要⼤量依赖实时整合的⽅式来进⾏主数据的集成和同步

    数据量不同:数据仓库存储的是⼤量的历史数据和各个维度的汇总数据，可能是海量的MDM存储的仅仅是客户和产品等信息主数据管理系统和数据仓库系统异同共存，互为补充。

主数据管理系统和ODS的关系

MDM和ODS存储的都是实时数据，但是MDM是不存储交易数据的，⼀般场景下，两个系统要实现客户信息和订单信息的共享，客户信息通过MDM共享，订单信息通过ODS或其他⼿段共享。

主数据管理典型应⽤

主数据管理系统的典型应⽤有客户管理与产品管理。

主数据管理在⾦融⾏业典型的应⽤就是企业级客户信息整合系统，其⽬标是整合全⾏现有业务系统中的客户信息，保留客户的最新信息，为个应⽤系统提供完整的、共享的、⼀致的客户信息，建⽴企业级客户单⼀视图。

基于元数据的主数据管理

元数据管理作为企业数据资产管理中的⼀项核⼼技术，也将为主数据管理提供有⼒⽀撑，具体表现在：

  1. 为业务实体的定义、关系和业务规则到IT实现之间提供清晰、标准的语义转换，提⾼业务和IT之间的⼀致性，保障IT系统能够真实反映业务事实
  2. 为主数据管理的各个数据处理阶段提供数据标准、数据映射关系和数据规则的描述，保障主数据管理的数据质量为主数据和其他系统之间的数据交互提供有关数据标准、数据映射关系和数据规则的描述
